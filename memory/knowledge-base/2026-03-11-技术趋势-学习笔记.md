## 二、Vue3/React最佳实践

### 2.1 Vue3最佳实践

**组合式函数（Composables）**
```typescript
// 封装可复用逻辑
export function useUser() {
  const user = ref(null)
  const loading = ref(false)
  
  const fetchUser = async (id: string) => {
    loading.value = true
    try {
      user.value = await api.getUser(id)
    } finally {
      loading.value = false
    }
  }
  
  return { user, loading, fetchUser }
}
```

**最佳实践清单**
- ✅ 使用 `<script setup>` 语法
- ✅ 优先使用 `ref` 而非 `reactive`
- ✅ 复杂逻辑抽离为 composables
- ✅ 使用 `defineProps`/`defineEmits` 类型定义
- ✅ 利用 `provide/inject` 跨层级传值
- ❌ 避免在 `v-for` 和 `v-if` 同时使用
- ❌ 不要直接修改 props

**性能优化**
- 使用 `shallowRef` 替代 `ref`（大数据对象）
- 使用 `markRaw` 跳过响应式转换
- 合理使用 `computed` 缓存计算
- 虚拟列表处理大量数据

### 2.2 React最佳实践

**Hooks最佳实践**
```typescript
// 自定义Hook封装业务逻辑
function useAsync<T>(asyncFunction: () => Promise<T>) {
  const [data, setData] = useState<T | null>(null)
  const [loading, setLoading] = useState(false)
  const [error, setError] = useState<Error | null>(null)
  
  const execute = useCallback(async () => {
    setLoading(true)
    try {
      const result = await asyncFunction()
      setData(result)
    } catch (e) {
      setError(e as Error)
    } finally {
      setLoading(false)
    }
  }, [asyncFunction])
  
  return { data, loading, error, execute }
}
```

**最佳实践清单**
- ✅ 函数组件 + Hooks
- ✅ 使用 TypeScript 严格类型
- ✅ 使用 React.memo 优化重渲染
- ✅ useMemo/useCallback 缓存昂贵计算
- ✅ 使用 Error Boundary 处理错误
- ❌ 避免在循环/条件中调用 Hooks
- ❌ 不要过度使用 useMemo

**状态管理选择**
| 场景 | 推荐方案 |
|------|----------|
| 本地组件状态 | useState/useReducer |
| 跨组件共享 | Context + useReducer |
| 服务端状态 | TanStack Query/SWR |
| 全局客户端状态 | Zustand/Jotai |
| 复杂应用 | Redux Toolkit |

### 2.3 全栈框架实践

**Next.js App Router**
```typescript
// Server Component - 服务端渲染
async function ProductPage({ params }: { params: { id: string } }) {
  const product = await getProduct(params.id) // 直接服务端获取数据
  
  return (
    <div>
      <h1>{product.name}</h1>
      <AddToCartButton productId={product.id} /> {/* Client Component */}
    </div>
  )
}
```

**关键特性**
- Server Components：服务端渲染，减少客户端JS
- Server Actions：服务端直接处理表单提交
- Parallel Routes：并行加载多个页面
- Intercepting Routes：模态框路由

**Nuxt 3 全栈**
- 自动导入：组件、composables自动识别
- 服务端路由：API路由内置
- 渲染模式：SSR/SSG/CSR灵活切换
- Nitro引擎：跨平台部署

| 工具 | 特点 | 适用场景 |
|------|------|----------|
| **GitHub Copilot** | 代码补全、聊天、多语言 | 日常编码、学习新API |

| **Tabnine** | 私有化部署 | 企业安全合规 |

### 3.2 AI辅助开发工作流

**代码生成**
```
1. 描述需求 → AI生成代码框架
2. 审查生成代码 → 调整优化
3. 补充测试 → AI生成测试用例
4. 代码审查 → AI辅助Review
```

**提示工程技巧**
- 提供上下文：相关代码、技术栈
- 明确约束：性能要求、兼容性
- 迭代优化：不满意就调整提示
- 验证输出：AI代码需要人工审查

### 3.3 AI工具使用原则

**DO**
- ✅ 用于重复性代码生成
- ✅ 学习新技术、API用法
- ✅ 代码解释、文档生成
- ✅ 测试用例生成
- ✅ 代码审查辅助

**DON'T**
- ❌ 完全依赖AI，不验证输出
- ❌ 处理敏感数据、安全代码
- ❌ 不经审查直接提交生产
- ❌ 放弃学习基础原理

## 四、云原生与DevOps

### 4.1 容器化与编排

**Docker最佳实践**
- 多阶段构建：减小镜像体积
- 非root用户运行：安全加固
- 健康检查：容器状态监控
- 镜像扫描：漏洞检测

**Kubernetes核心概念**
```
Pod → 最小部署单元
Deployment → 管理Pod副本
Service → 服务发现与负载均衡
Ingress → 外部访问入口
ConfigMap/Secret → 配置管理
PersistentVolume → 持久化存储
```

**K8s工具链**
- **Helm**：包管理器
- **Kustomize**：配置定制
- **ArgoCD**：GitOps持续交付
- **Prometheus + Grafana**：监控告警

### 4.2 CI/CD流水线

**GitHub Actions工作流**
```yaml
name: CI/CD Pipeline
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run test
      - run: npm run build
  
  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploy to production"
```

**CI/CD最佳实践**
- 自动化测试：单元测试、集成测试、E2E测试
- 代码质量：ESLint、Prettier、SonarQube
- 安全扫描：依赖漏洞、密钥检测
- 蓝绿部署/金丝雀发布：零停机发布

## 五、低代码/无代码平台

### 5.1 平台类型

| 类型 | 代表产品 | 适用场景 |
|------|----------|----------|
| 应用搭建 | 钉钉宜搭、飞书多维表格 | 内部工具、表单系统 |
| 流程自动化 | Zapier、Make、飞书捷径 | 工作流自动化 |
| 网站构建 | Webflow、Framer | 营销网站、落地页 |
| 数据库应用 | Airtable、Notion | 数据管理、轻量CRM |
| 全栈开发 | Retool、Appsmith | 内部后台系统 |

### 5.2 低代码适用场景

**推荐使用**
- ✅ 内部管理后台
- ✅ 数据收集表单
- ✅ 简单工作流
- ✅ MVP快速验证
- ✅ 运营配置页面

**不推荐**
- ❌ 复杂业务逻辑
- ❌ 高性能要求场景
- ❌ 强定制化UI
- ❌ 核心交易系统

### 5.3 低代码与传统开发结合

**混合架构**
```
前端展示层：低代码平台搭建
业务逻辑层：传统API服务
数据存储层：数据库/数据仓库
```

**最佳实践**
- 低代码负责UI和简单逻辑
- 复杂逻辑下沉到后端服务
- API网关统一接口管理
- 数据同步机制保障一致性
