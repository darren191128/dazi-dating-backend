## 一、产品需求文档（PRD）撰写规范

### 1.1 PRD核心结构

```
1. 文档信息
   - 文档标题、版本号、作者、日期
   - 变更记录（版本历史）

2. 背景与目标
   - 项目背景：为什么要做这个产品/功能
   - 产品目标：要解决的问题、预期成果
   - 成功指标：如何衡量成功

3. 用户场景
   - 目标用户画像
   - 用户故事（User Stories）
   - 使用场景描述

4. 功能需求
   - 功能清单
   - 详细功能描述
   - 业务流程图
   - 页面原型/交互说明

5. 非功能需求
   - 性能要求
   - 安全要求
   - 兼容性要求

6. 数据埋点
   - 需要采集的数据
   - 埋点位置说明

7. 附录
   - 参考文档
   - 术语表
```

### 1.2 用户故事（User Story）规范

**标准格式**
```
作为 [用户角色]，
我想要 [功能/能力]，
以便 [获得的价值/目的]。
```

**INVEST原则**
- **I**ndependent（独立的）：可独立开发交付
- **N**egotiable（可协商的）：细节可与团队讨论
- **V**aluable（有价值的）：对用户有明确价值
- **E**stimable（可估算的）：开发团队能估算工作量
- **S**mall（小的）：一个迭代内可完成
- **T**estable（可测试的）：有明确的验收标准

**验收标准（Acceptance Criteria）**
```
Given [前置条件]
When [操作/触发]
Then [预期结果]
```

### 1.3 PRD撰写最佳实践

**DO（推荐做法）**
- 从用户视角描述需求
- 使用具体、可量化的描述
- 配合流程图、原型图说明
- 明确优先级（P0/P1/P2）
- 保持文档实时更新

**DON'T（避免做法）**
- 过度描述技术实现细节
- 使用模糊词汇（"可能"、"大概"）
- 一次性写完美再评审
- 忽视边界情况和异常流程

## 二、用户研究与需求分析方法

### 2.1 用户研究方法

**定性研究**
| 方法 | 适用场景 | 样本量 | 输出 |
|------|----------|--------|------|
| 用户访谈 | 深度了解用户需求 | 5-10人 | 用户洞察、痛点 |
| 焦点小组 | 探索群体观点 | 6-10人/组 | 群体趋势 |
| 可用性测试 | 验证产品易用性 | 5-8人 | 可用性问题 |
| 情境访谈 | 观察用户实际行为 | 3-5人 | 真实使用模式 |

**定量研究**
| 方法 | 适用场景 | 样本量 | 输出 |
|------|----------|--------|------|
| 问卷调查 | 验证假设、量化需求 | 100+ | 统计数据 |
| A/B测试 | 对比方案效果 | 流量决定 | 转化率数据 |
| 数据分析 | 了解用户行为 | 全量 | 行为模式 |
| 漏斗分析 | 识别流失点 | 全量 | 转化瓶颈 |

### 2.2 需求分析框架

**5W2H分析法**
- **W**hat：用户需要什么？
- **W**hy：为什么需要？背后的动机？
- **W**ho：谁有这个问题？
- **W**hen：什么时候需要？
- **W**here：在什么场景下？
- **H**ow：用户目前如何解决？
- **H**ow much：用户愿意付出什么代价？

**Jobs-to-be-Done（JTBD）**
- 核心观点：用户"雇佣"产品完成特定"工作"
- 关注功能需求背后的情感和社会需求
- 问题框架：当____，我想要____，以便我能____

### 2.3 需求优先级评估

**Kano模型**
```
        满意度
          ↑
  兴奋型  │    /
  需求    │   /
          │  /
  期望型  │ /
  需求    │/
          └──────────→ 功能实现程度
         /│
        / │  基本型
       /  │  需求
      /   │
```

- **基本型需求**：必须有，没有会极度不满
- **期望型需求**：有则满意，没有则不满
- **兴奋型需求**：没有无所谓，有了会惊喜

**价值 vs 复杂度矩阵**
```
        高价值
           │
  快速获胜 │  重点项目
  (先做)   │  (投入资源)
           │
  ─────────┼────────→ 高复杂度
           │
  低优先级 │  填坑项目
  (不做)   │  (考虑不做)
           │
        低价值
```

## 三、产品原型设计原则

### 3.1 原型保真度选择

| 保真度 | 适用阶段 | 工具 | 特点 |
|--------|----------|------|------|
| 低保真 | 概念验证 | 纸笔、Balsamiq | 快速、低成本 |
| 中保真 | 方案讨论 | Figma、Axure | 结构清晰、可交互 |
| 高保真 | 最终确认 | Figma、Sketch | 接近真实、开发参考 |

### 3.2 设计原则

**尼尔森十大可用性原则**
1. **系统状态可见性**：让用户知道发生了什么
2. **系统与现实世界匹配**：使用用户熟悉的语言
3. **用户控制与自由**：支持撤销和重做
4. **一致性与标准**：遵循平台惯例
5. **错误预防**：设计防止错误发生
6. **识别而非回忆**：减少用户记忆负担
7. **灵活性与效率**：兼顾新手和专家用户
8. **美学与简约设计**：不相关的内容不显示
9. **帮助用户识别、诊断和恢复错误**：清晰的错误提示
10. **帮助与文档**：易于搜索、步骤清晰

**移动端设计原则**
- 拇指友好：重要操作在拇指可及区域
- 减少输入：使用选择器、自动填充
- 内容优先：减少导航层级
- 即时反馈：操作后立即响应
- 容错设计：误操作可恢复

### 3.3 原型评审要点

**功能完整性**
- 主流程是否完整？
- 异常流程是否考虑？
- 边界情况如何处理？

**用户体验**
- 操作是否直观？
- 反馈是否清晰？
- 学习成本如何？

**技术可行性**
- 开发成本评估
- 技术限制确认
- 第三方依赖

## 四、MVP与产品迭代策略

### 4.1 MVP（最小可行产品）

**定义**
- 用最小成本、最短时间做出能验证核心假设的产品
- 不是半成品，而是能解决核心问题的最小功能集

**MVP设计原则**
1. 聚焦核心问题：解决最痛的那个点
2. 手动优先：能人工做的不用自动化
3. 借用现有工具：不重复造轮子
4. 快速上线：2-4周内发布

**MVP常见误区**
- ❌ MVP = 粗糙的产品
- ❌ 功能越多越好
- ❌ 追求完美再发布
- ✅ MVP = 刚好能验证假设的产品

### 4.2 产品迭代策略

**迭代节奏**
| 周期 | 活动 | 输出 |
|------|------|------|
| 每周 | 数据分析、用户反馈收集 | 问题清单 |
| 每双周 | Sprint规划、开发、发布 | 功能更新 |
| 每月 | 版本回顾、 roadmap调整 | 下月计划 |
| 每季 | 大版本规划、OKR复盘 | 季度规划 |

**迭代决策依据**
- 数据指标：转化率、留存率、活跃度
- 用户反馈：客服工单、应用商店评论、用户访谈
- 业务目标：营收、市场份额、战略目标
- 技术债务：系统稳定性、性能优化

### 4.3 增长模型

**AARRR漏斗**
- **A**cquisition（获取）：用户从哪里来？
- **A**ctivation（激活）：用户首次体验如何？
- **R**etention（留存）：用户是否持续使用？
- **R**eferral（推荐）：用户是否愿意推荐？
- **R**evenue（收入）：如何变现？

**增长实验流程**
1. 提出假设：如果我们做X，Y指标会提升Z%
2. 设计实验：MVP方案、测试方法
3. 执行实验：小范围测试
4. 分析结果：数据验证假设
5. 决策：扩大规模/优化/放弃

## 五、竞品分析方法

### 5.1 竞品选择

**直接竞品**：解决相同问题、目标用户相同
**间接竞品**：解决相同问题、方式不同
**潜在竞品**：可能进入市场的玩家
**标杆产品**：行业最佳实践参考

### 5.2 分析维度

**产品层面**
| 维度 | 分析内容 |
|------|----------|
| 功能对比 | 功能清单、差异化功能 |
| 用户体验 | 交互流程、视觉设计 |
| 技术实现 | 性能、稳定性 |
| 商业模式 | 定价策略、变现方式 |

**市场层面**
| 维度 | 分析内容 |
|------|----------|
| 用户规模 | 下载量、活跃用户数 |
| 用户评价 | 评分、评论分析 |
| 市场份额 | 市场占有率 |
| 增长趋势 | 增长率、发展态势 |

**战略层面**
| 维度 | 分析内容 |
|------|----------|
| 定位策略 | 目标用户、价值主张 |
| 发展路线 | 产品迭代方向 |
| 资源投入 | 团队规模、融资情况 |

### 5.3 竞品分析框架

**SWOT分析**
- **S**trengths（优势）：竞品做得好的地方
- **W**eaknesses（劣势）：竞品的不足
- **O**pportunities（机会）：市场空白点
- **T**hreats（威胁）：竞争压力

**竞品画布**
```
┌─────────────────┬─────────────────┐
│   竞品名称       │   我们的定位     │
├─────────────────┼─────────────────┤
│ 目标用户         │ 目标用户         │
├─────────────────┼─────────────────┤
│ 核心功能         │ 核心功能         │
├─────────────────┼─────────────────┤
│ 差异化优势       │ 差异化优势       │
├─────────────────┼─────────────────┤
│ 商业模式         │ 商业模式         │
└─────────────────┴─────────────────┘
```

### 5.4 分析工具

- **SimilarWeb**：网站流量分析
- **App Annie / Sensor Tower**：App数据分析
- **SEMrush**：SEO和广告分析
- **社交媒体**：用户评价、舆情监测
- **亲自体验**：深度使用竞品产品

## 七、核心要点总结

1. **PRD撰写**：结构清晰、用户视角、验收标准明确
2. **用户研究**：定性+定量结合，JTBD深挖需求本质
3. **原型设计**：根据阶段选择保真度，遵循可用性原则
4. **MVP与迭代**：快速验证、数据驱动、持续优化
5. **竞品分析**：多维度对比，找到差异化机会
