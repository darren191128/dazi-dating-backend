# 项目执行问题记录

## 问题概述
**时间**：2026-03-12  
**项目**：若山律师事务所官网重构  
**问题**：云肆（UI设计）任务超时

## 问题详情

### 超时情况
- **任务**：UI设计（创建设计标准 + 7个页面HTML）
- **预计时间**：3小时
- **实际执行**：2次尝试均超时
- **超时设置**：10分钟

### 根本原因分析

1. **任务粒度过大**
   - 一次性要求完成：设计标准文档 + 7个页面 + 后台CMS
   - 工作量超出单次子智能体合理执行范围
   - 子智能体在生成过程中被中断

2. **超时设置不合理**
   - 默认超时10分钟对于UI设计任务过短
   - 复杂页面设计需要更长时间

3. **缺乏进度保存机制**
   - 子智能体无法在超时前保存中间成果
   - 每次重启都从零开始

4. **模型特性问题**
   - 云肆使用Kimi模型，代码生成速度相对较慢
   - 同时生成多个HTML页面导致token消耗过大

## 解决方案

### 短期（本次项目）
1. ✅ 改为并行策略：云伍/云陆直接基于PRD开发
2. ✅ UI设计后续补充或由前端直接实现

### 长期（后续项目）
1. **任务拆分原则**
   - UI设计按页面拆分：首页、关于、服务、团队、案例、新闻、联系、后台
   - 每个页面单独一个子任务
   - 设置合理超时：单页面20-30分钟

2. **增加检查点机制**
   - 每完成一个页面立即保存
   - 定期汇报进度
   - 支持断点续传

3. **模型分配优化**
   - UI设计可考虑使用更快的模型
   - 或增加超时时间到30分钟以上

4. **前置条件确认**
   - 确保设计文档完整后再启动开发
   - 避免开发等待设计的情况

## 改进后的流程

```
阶段1：项目分析（云捌-GLM）
  ↓
阶段2：UI设计（云肆-Kimi）拆分为8个子任务
  - 设计标准（单独任务）
  - 首页（单独任务）
  - 关于我们（单独任务）
  - ...其他页面
  ↓
阶段3：审核（云伍+云陆）
  ↓
阶段4：开发（云伍+云陆并行）
```

## 经验教训

1. **不要贪多**：子智能体任务要小而精
2. **设置合理超时**：根据任务复杂度调整
3. **及时保存成果**：避免重复劳动
4. **并行优于串行**：减少等待时间
