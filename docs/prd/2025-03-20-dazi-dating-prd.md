# 搭子交友微信小程序 PRD

## 1. Executive Summary

搭子交友是一款基于地理位置和兴趣匹配的社交小程序，帮助用户找到附近的"搭子"（饭搭子、运动搭子、旅游搭子等）一起参与线上或线下活动。产品核心特色是智能匹配算法（兴趣>位置>星座/生肖>活动类型>年龄/性别）和灵活的费用分摊机制（AA制、男A女免、请客、免费），同时提供完善的安全保障体系（实名认证+信用分+双向评价+举报机制）。

## 2. Problem Statement

**Who:** 18-45岁都市人群，包括职场新人、大学生、宝妈、户外爱好者等

**What:** 想要找人一起参加活动（吃饭、运动、旅游等），但缺乏合适的渠道

**Why it's painful:**
- 朋友圈有限，难以找到志同道合的伙伴
- 现有社交平台匹配不精准，效率低下
- 活动费用分摊不透明，容易产生纠纷
- 担心陌生人社交的安全问题
- 缺乏专门的活动发布和参与平台

**Evidence:**
- "找饭搭子""找运动搭子"等关键词在小红书、微博搜索量持续增长
- 豆瓣同城、陌陌等平台的兴趣小组活跃度高
- 线下活动组织分散，缺乏统一平台

## 5. Solution Overview

### Core Features

#### 1. 智能匹配系统
- **匹配维度**: 兴趣标签(40%) + 地理位置(30%) + 星座/生肖(15%) + 活动类型(10%) + 年龄/性别(5%)
- **推荐算法**: 基于用户画像的综合评分排序
- **筛选功能**: 支持按距离、兴趣、活动类型筛选

#### 2. 活动系统
**活动类型:**
- ● 线下活动（默认）
- ○ 线上活动

**线下活动分类:**
- 🍜 吃喝玩乐（聚餐、K歌、看电影）
- 🏕️ 户外游玩（郊游、旅游、露营）
- 👶 亲子活动（儿童乐园、亲子游）
- 🏃 户外运动（羽毛球、篮球、跑步）
- 💕 相亲交友（单身聚会、联谊活动）

**线上活动分类:**
- 🎮 游戏开黑（王者荣耀、吃鸡、LOL等）
- 🎤 线上K歌（全民K歌、唱吧）
- 📺 观影追剧（腾讯会议一起看）
- 💬 语音聊天（情感倾诉、话题讨论）
- 📚 学习打卡（自习室、读书会）

**费用类型:**
- 💰 AA制：总金额÷人数
- 👨💰👩🆓 男A女免：男性分摊全部费用，女性免费
- 🎁 请客：发起人支付全部费用
- 🆓 免费：无需支付费用

**资金托管与支付流程:**
```
1. 活动发布：
   - 发起人发布活动（选择费用类型）
   - 付费活动需平台审核通过后上线

2. 用户报名：
   - 查看活动详情（费用明细透明展示）
   - 点击报名 → 确认费用 → 调起微信支付
   - 资金进入平台托管账户（非直接给发起人）

3. 资金托管：
   - 用户支付 → 平台托管账户
   - 活动开始前24小时：资金解冻，可提现
   - 活动完成：发起人发起提现申请
   - T+1到账（扣除1%平台服务费+0.6%支付手续费）

4. 退款机制：
   - 活动取消（发起人）：全额退款给所有参与者
   - 取消报名（用户）：活动前24小时可全额退款
   - 活动前24小时内取消：扣除20%违约金给发起人
   - 纠纷处理：平台介入，冻结资金直至解决

5. 安全保障：
   - 平台不触碰资金，仅做托管
   - 资金流水透明可查
   - 提现需实名认证+银行卡验证
```

#### 3. 安全保障体系
- **实名认证**: 身份证+人脸识别
- **信用分系统**: 0-1000分，基于活动参与记录和评价
- **双向评价**: 活动后双方互评
- **举报机制**: 快速处理违规用户
- **紧急求助**: 一键报警+位置共享

**信用分计算规则:**
```
加分项：
+ 完成实名认证：+100分
+ 完成首次活动：+50分
+ 获得好评：+20分/次
+ 连续参加活动：+10分/次（每月上限50）
+ 完善个人资料：+30分

减分项：
- 获得差评：-30分/次
- 取消报名（活动前2小时内）：-20分/次
- 被举报核实：-50分/次
- 放鸽子（报名后未参加）：-100分/次
- 发布违规内容：-80分/次

等级划分：
- 青铜：0-300分（限制发布付费活动）
- 白银：301-500分
- 黄金：501-700分
- 铂金：701-900分
- 钻石：901-1000分（优先推荐）
```

**反作弊机制:**
```
1. 新用户限制：
   - 注册7天内不能发布付费活动
   - 注册3天内不能发起超过2个活动
   - 新用户报名需信用分>300

2. 活动审核：
   - 付费活动需平台人工审核（工作时间内2小时内）
   - 免费活动AI自动审核（敏感词+图片识别）
   -  suspicious活动（高金额、异常描述）人工复核

3. 设备指纹：
   - 检测多账号登录（同一设备限制3个账号）
   - 虚拟机/模拟器检测
   - 异常IP地址标记

4. 行为分析：
   - 秒报名检测（报名间隔<5秒视为异常）
   - 高频发布检测（同一用户1天内发布>3个活动）
   - 资金异常检测（短时间内大额资金流动）

5. 处罚措施：
   - 首次违规：警告+扣除信用分
   - 二次违规：限制功能7天
   - 三次违规：永久封号
```

**隐私合规措施:**
```
1. 隐私政策：
   - 独立的隐私政策页面
   - 明确说明数据收集范围：位置、身份证、人脸、聊天记录
   - 说明数据使用目的：匹配推荐、安全保障、支付结算
   - 说明数据存储期限：注销后30天删除

2. 用户授权：
   - 首次使用需同意《用户协议》和《隐私政策》
   - 敏感权限单独授权（位置、相机、相册、麦克风）
   - 可随时撤回授权（设置页面）

3. 数据保护：
   - 身份证和人脸数据加密存储（AES-256）
   - 位置数据模糊处理（精确到500米）
   - 聊天记录本地存储，服务器不保存
   - 数据传输全程HTTPS加密

4. 用户权利：
   - 查看个人数据（我的-数据管理）
   - 导出个人数据（申请后24小时内提供）
   - 删除账号（注销后30天彻底清除数据）
   - 撤回同意（撤回后停止使用相关功能）

5. 未成年人保护：
   - 实名认证时检测年龄
   - 未满18岁需监护人同意
   - 未成年人限制部分功能（如相亲交友）
```

**内容审核机制:**
```
1. AI自动审核（实时）：
   - 敏感词过滤（政治、色情、暴力、诈骗）
   - 图片鉴黄（阿里云/腾讯云内容安全）
   - 文本分类（广告、垃圾信息识别）
   - 命中敏感词自动拦截，人工复核

2. 人工审核（2小时内）：
   - 付费活动必审
   - 免费活动抽审（10%）
   - 被举报内容优先审核
   - 审核结果：通过/驳回/需修改

3. 用户举报：
   - 一键举报（活动/用户/聊天）
   - 举报类型：虚假信息、诈骗、骚扰、违规内容
   - 2小时内响应，24小时内处理
   - 举报人收到处理结果通知

4. 审核日志：
   - 记录所有审核操作
   - 记录所有违规处理
   - 支持审计追溯
```

#### 4. 即时通讯
- 活动群内聊天
- 私聊功能
- 活动通知推送

## 11. 运营后台功能

### 后台管理系统（Web端）

#### 1. 用户管理

## 17. 性能优化策略

```
分库分表（未来）：
- 用户量超100万：按user_id % 10分表
```

## 18. 安全加固

### 接口安全
```
传输安全：
- HTTPS全站加密（TLS 1.3）
- HSTS头设置
- 证书定期更新（自动续期）

请求安全：
- 接口签名校验：timestamp + nonce + sign
- 防重放攻击：timestamp有效期5分钟，nonce唯一性校验
- CSRF防护：SameSite Cookie + Token验证
- CORS限制：只允许小程序域名访问
```

### 数据安全
```
存储安全：
- 密码加密：BCrypt（cost=12）
- 敏感数据加密：AES-256（身份证、手机号）
- 密钥管理：使用阿里云KMS

传输安全：
- 敏感信息脱敏：日志中手机号显示138****8888
- 防止信息泄露：错误信息不暴露内部细节

防护机制：
- SQL注入：MyBatis参数化查询，禁止拼接SQL
- XSS防护：输入过滤 + 输出转义
- 文件上传：限制类型、大小，病毒扫描
```

### 业务安全
```
账号安全：
- 登录失败锁定：5次失败锁定30分钟
- 异地登录提醒：检测到异常地点登录发送通知
- 设备管理：查看登录设备，可踢出其他设备

反欺诈：
- 防刷单：同一设备限制3个账号
- 防羊毛党：新用户奖励限制（需完成首次活动）
- 防爬虫：接口频率限制 + 行为分析
- 防社工：客服不接触用户密码、支付密码

资金安全：
- 支付密码：与登录密码分离
- 大额交易确认：超过500元需二次确认
- 资金变动通知：每笔资金变动发送通知
```

### 审计日志
```
记录内容：
- 用户登录：时间、IP、设备、结果
- 敏感操作：修改密码、绑定手机、实名认证
- 资金操作：充值、提现、支付、退款
- 管理操作：后台管理员的所有操作

日志管理：
- 日志格式：JSON格式，包含时间、用户、操作、结果
- 日志存储：保存180天，定期归档
- 日志分析：ELK Stack分析异常行为
- 审计报告：每月生成安全审计报告
```

## 21. 灾备与容灾

### 数据备份策略
```
备份类型：
- 全量备份：每日凌晨3:00自动执行
- 增量备份：实时binlog同步到备份服务器
- 冷备份：每周日全量备份到异地（OSS）

备份保留：
- 本地备份：保留30天
- 异地备份：保留90天
- 归档备份：保留1年（重要数据）

备份验证：
- 每周随机抽取备份进行恢复测试
- 验证数据完整性（校验和比对）
- 验证恢复时间（RTO < 30分钟）
```

### 服务容灾
```
多可用区部署：
- ECS：跨2个可用区部署（主+备）
- 负载均衡：自动流量切换
- 数据库：主从架构，自动故障切换
- Redis：哨兵模式，自动选举主节点

故障切换：
- 数据库故障：10秒内切换到从库
- 服务故障：5秒内剔除故障节点
- 缓存故障：直接查询数据库（降级）
- 第三方故障：启用降级方案

降级策略：
- 匹配服务故障：显示缓存数据，提示"推荐服务维护中"
- 支付服务故障：仅支持免费活动，暂停付费功能
- 地图服务故障：使用IP定位，精度降低
- 审核服务故障：暂停UGC功能，仅展示历史内容
```

### 应急预案
```
故障分级：
- P0（严重）：服务完全不可用，影响所有用户
- P1（高）：核心功能不可用，影响部分用户
- P2（中）：非核心功能异常
- P3（低）：轻微异常，不影响使用

响应时间：
- P0：5分钟内响应，30分钟内恢复
- P1：15分钟内响应，1小时内恢复
- P2：1小时内响应，4小时内恢复
- P3：4小时内响应，24小时内恢复

应急联系人：
- 技术负责人：电话 + 微信
- 运维负责人：电话 + 微信
- 产品负责人：电话 + 微信
- 客服负责人：电话 + 微信

故障演练：
- 每月一次故障演练
- 模拟场景：数据库宕机、服务OOM、网络中断
- 演练目标：验证恢复流程，锻炼团队响应能力
- 演练报告：记录问题，持续改进
```

## 23. AB测试方案

### 测试框架
```
分组策略：
- 用户分组：user_id % 100
- 对照组：0-49（50%）
- 实验组：50-99（50%）
- 支持多实验并行（正交分层）

   - 对照组：兴趣40% + 位置30% + 星座15% + 活动10% + 年龄5%
   - 实验组：兴趣50% + 位置25% + 星座10% + 活动10% + 年龄5%
   - 指标：匹配成功率、用户满意度

2. UI设计：
   - 对照组：红色主题（#ff6b6b）
   - 实验组：蓝色主题（#4a90e2）
   - 指标：页面停留时长、点击率

   - 对照组："找搭子，一起吃喝玩乐"

   - 指标：注册转化率、次日留存
```

## 25. 版本发布策略

### 版本管理
```
版本号规则：主版本.次版本.修订号
- 主版本：重大架构变更（如V1→V2）
- 次版本：新增功能（如V1.1→V1.2）
- 修订号：Bug修复（如V1.1.0→V1.1.1）

版本示例：
- V1.0.0：初始版本（MVP）
- V1.1.0：新增语音聊天功能
- V1.1.1：修复支付Bug
- V1.2.0：新增运营后台
- V2.0.0：架构升级，支持App
```

### 发布流程
```
1. 开发阶段（Day 1-20）：
   - 功能开发
   - 单元测试
   - 代码审查

2. 测试阶段（Day 21-25）：
   - QA测试
   - Bug修复
   - 回归测试

3. 预发布阶段（Day 26-27）：
   - 部署到预发布环境
   - 内部试用
   - 性能测试

4. 灰度发布（Day 28）：
   - 5%用户（约500人）
   - 监控核心指标
   - 收集用户反馈

5. 全量发布（Day 29）：
   - 100%用户
   - 持续监控24小时
   - 准备回滚方案

6. 观察期（Day 30）：
   - 观察7天
   - 收集用户反馈
   - 修复线上问题
```

### 回滚机制
```
回滚触发条件：
- P0故障：服务完全不可用
- 严重Bug：影响核心功能，大量用户投诉
- 数据错误：资金计算错误、数据丢失
- 安全漏洞：被黑客攻击，数据泄露

回滚流程：
1. 发现故障（5分钟内）
2. 确认需要回滚（10分钟内）
3. 执行回滚（保留当前版本镜像）
4. 切换到上一个稳定版本（5分钟内）
5. 验证服务恢复（10分钟内）
6. 通知用户（发布服务公告）
7. 事后复盘（24小时内）

回滚时间：
- 目标：从发现故障到恢复服务 < 30分钟
- 实际：根据演练结果持续优化
```
